# ADR 0012: Phase 7.5 范围 + 4 子阶段切片

- **Status**: Accepted
- **Date**: 2026-05-16
- **Deciders**: project lead
- **Supersedes**: 无
- **Superseded by**: 无

## Context

PROJECT_SPEC.md §3.11 / §3.11.1 / §3.11.2 + §10 + Phase 7.5 deliverables 列出了一组规模相当大的产物：

- **v1.1 prediction_registry/**：registry / snapshot 升级 PG，outcome_tracker / event_detector / attribution / review_workflow / benchmarks 5 个完整实现（Phase 6 留下都是 4 行 docstring 占位）
- **v1.2 ipo_lifecycle/**：states / state_machine / state_detectors（三重 LISTED 验证）/ stale_detector / terminal_handlers / ah_special 共 6 个模块
- **v1.2 schedulers/**：base / high_frequency / daily / event_driven + advisory lock + Airflow DAG ≥4
- **v1.2 code_mapper / earnings_comparator / alerts** + `config/alerts.yaml` + `config/mapping_rules.yaml`
- **DB schema**：v1.1 (4 表 + 1 trigger 函数 + 2 trigger) + v1.2 (6 表) + v1.2.1 (8 表 + 1 trigger 复用) = 共 **18 张表 + 4 个 trigger** 需要在 Phase 2 migration 基础上一次落地
- **orchestrator 调整**：`synthesize → create_snapshot(PG) → report` 强制硬边，create_snapshot 失败 → 整流程失败
- **ADR 0011 遗留**：Phase 7 的 in-memory audit_log / users / chat / whatif / realtime_events 替换为 PG + DB trigger；`api/routers/{reviews, proposals, drift}.py` 三个 stub 实装

按 CLAUDE.md「函数 ≤ 50 行 / 文件 ≤ 500 行」+ 实证 Phase 5/6/7 节奏，全量实施约 ~7 天，spec 估 6-7 天。需要明确切片以保证：

1. 每个子阶段独立 commit + 单测 + 停下来等确认（符合 memory 中的 Phase 工作流偏好）
2. 端到端 DONE 条件（晶泰 2228.HK lifecycle 模拟）放最后一子阶段，前面三阶段不被它阻塞
3. ADR 0011 遗留的两条 Phase 7.5 Progress 行有明确接力子阶段，能在切片末尾勾完

参考 ADR 0011（Phase 7 范围切片）模式 — 但本 ADR 不切 MVP / 延期，因为 Phase 7.5 本身就是 spec §3.11 自包含，没有可延期的项。

## Decision

**Phase 7.5 切成 4 个子阶段 7.5a → 7.5b → 7.5c → 7.5d，每个子阶段独立 commit + 测试 + 停下来等用户确认；最后一子阶段完结后打 tag `v0.7.5`。**

切片原则：
- **7.5a 先把所有 DB 表 + Registry PG 化一次性落地**：避免两轮 Alembic migration；让 7.5b/c/d 在稳定的 PG 层上编码
- **7.5b 闭合 v1.1 数据流**：outcome / event / attribution / review 是同一条预测追踪流水线，必须连一起测才有意义
- **7.5c 闭合 v1.2 业务规则**：状态机 + 代码映射 + 财报比对 + 警报路由是「单 IPO 实例」层级的自动化
- **7.5d 闭合 v1.2 调度引擎 + 端到端**：调度器是系统能自主运行的「真正引擎」(spec 原话)，必须最后落地并跑端到端晶泰验证

| `prediction_registry/outcome_tracker.py` | outcome_tracker.py | T+N checkpoint 拉股价（iFind `as_of_date` 模式） + 3 基准超额收益 + 调用 event_detector + 幂等 (snapshot_id, checkpoint_day) UNIQUE |

- OpenAPI 3.1 schema 含 reviews/proposals/drift；UI `openapi-typescript` 能生成完整 TS 类型
- **ADR 0011 Progress 行的两条 Phase 7.5 条目（"替换 in-memory" + "实装 reviews/proposals/drift"）勾掉**

### Phase 7.5c — IPO 状态机 + 代码映射 + 财报比对 + 警报（~1.5 天）

**范围**：

| 子项 | 文件 | 关键功能 |
|---|---|---|
| `prediction_registry/ipo_lifecycle/states.py` | states.py | 7 状态 enum（PRE_LISTING / PRICING / LISTED / WITHDRAWN / HEARING_FAILED / PRICING_PULLED / TERMINATED） + `VALID_TRANSITIONS` 表 + `InvalidStateTransition` 异常 |
| `prediction_registry/ipo_lifecycle/state_machine.py` | state_machine.py | `get_state` / `transition_to`（严格走 VALID_TRANSITIONS） + 每次 transition 写 `ipo_state_transitions` audit log |
| `prediction_registry/ipo_lifecycle/state_detectors.py` | state_detectors.py | PRICING / **LISTED 三重验证**（HKEX 公告 + iFind 首日行情 + 股票代码激活） / WITHDRAWN / HEARING_FAILED 各自检测函数 |
| `prediction_registry/ipo_lifecycle/stale_detector.py` | stale_detector.py | PRE_LISTING >180d（招股书 6 个月有效期失效） + PRICING >21d；警报但不自动转 WITHDRAWN |
| `prediction_registry/ipo_lifecycle/terminal_handlers.py` | terminal_handlers.py | 撤回 / 聆讯失败 → 生成 `terminal_review_draft`（聚焦"漏看了什么信号"） + 写 `prediction_outcomes(checkpoint_day=-1)` 标记终态 |
| `prediction_registry/ipo_lifecycle/ah_special.py` | ah_special.py | A+H：上市日 = H 股首日；checkpoint 计时从 H 股起；归因时考虑 A 股期间表现已消化的部分信息 |
| `prediction_registry/code_mapper.py` | code_mapper.py | 3 策略优先级（HKEX 公告 → iFind search_by_name → 保荐 + 日期反查）+ 置信度 high/medium/low + **low → `requires_review=True` + 警报** + A+H 同时映射 H/A 代码 |
| `prediction_registry/earnings_comparator.py` | earnings_comparator.py | event_detector → earnings 触发 → `compare(snapshot_id, filing)` → 5 维度对比 → 写 `earnings_comparisons` + 前 3 次强制 `requires_human_review=True` |
| `prediction_registry/alerts.py` | alerts.py | 3 level 路由（info → log / warning → 24h review / critical → 立即通知） + 渠道（Email / Slack / PagerDuty / SMS） + 24h `(category, ipo_id, level)` 去重 + **`actionable_info` 强制非空** |
| `config/alerts.yaml` | config | 渠道配置 + level 路由规则 |
| `config/mapping_rules.yaml` | config | 财报口径映射规则（按公司类型：AI / 半导体 / SaaS / 18C / 18A 等） |

**DONE 条件**：
- **3 个状态机仿真测试通过**：
  - 仿真"正常上市"：PRE_LISTING → PRICING → LISTED → 11 checkpoint → TERMINATED
  - 仿真"撤回"：PRE_LISTING → WITHDRAWN → 自动生成 terminal_review_draft
  - 仿真"默默失效"：PRE_LISTING 停留 181 天 → stale_detector 触发 critical alert
- **代码映射准确率测试**：30 家历史已上市公司 fixture，high confidence 映射准确率 ≥ 95%
- **财报比对测试**：5 家公司 PHIP 招股书 + 上市后首份年报 fixture，自动比对结果与人工比对吻合（容忍范围 ±5pp）
- 警报去重单测：同 (category, ipo_id, level) 在 24h 窗口内只发一次
- 状态机回退对抗测试：尝试 LISTED → PRE_LISTING 必抛 `InvalidStateTransition`
- 三重 LISTED 验证对抗测试：任一缺失（HKEX/iFind/code）均不转 LISTED

### Phase 7.5d — 三层调度器 + Airflow DAG + 端到端晶泰（~1.5 天）

**范围**：

| 子项 | 文件 | 关键功能 |
|---|---|---|
| `prediction_registry/schedulers/base.py` | base.py | `BaseScheduler` 抽象 + `scheduler_runs` 表写入（run_id / status / processed / errors） + **DB advisory lock 防重叠** + 失败重试 + 升级警报 |
| `prediction_registry/schedulers/high_frequency_scheduler.py` | high_frequency_scheduler.py | 每 15-30 min — 扫 active snapshot 状态变化 → state_detectors → code_mapper（LISTED 时） + event_detector(lookback="2h")；**不做归因/回测** |
| `prediction_registry/schedulers/daily_scheduler.py` | daily_scheduler.py | 每天 02:00-03:00 — checkpoint 触达 → outcome_tracker；补漏机制（扫历史未记录 checkpoint 用 historical close 补跑）；大 checkpoint (30/90/180/360) 触发 review_workflow；stale_detector；360d → TERMINATED 归档 |
| `prediction_registry/schedulers/event_driven_scheduler.py` | event_driven_scheduler.py | HKEX 公告流（RSS / webhook / 轮询）+ iFind 异动 + 披露易持股变动 → 路由到 earnings_comparator / cornerstone_tracker / alerts |
| `prediction_registry/schedulers/airflow_dags/` | 4 DAG | `daily_outcome_tracking.py`（SLA 监控 6h） + `high_freq_state_scan.py` + `event_driven_handler.py` + `earnings_comparison_trigger.py` |
| `config/schedulers.yaml` | config | cron 表达式 + 重试策略 + 并发度 + advisory lock 名 |
| 端到端晶泰测试 | `tests/e2e/test_phase_7_5_full_lifecycle_jingtai.py` | 用晶泰 2228.HK 作完整 lifecycle 模拟 |

**DONE 条件**：
- **调度器对抗测试**：
  - 重复运行同 checkpoint 必须幂等（snapshot_id, checkpoint_day UNIQUE 拦截）
  - 调度器中断后重启从 last successful run 恢复（`scheduler_runs.status='failed'` → 重跑该 run）
  - 错过的 checkpoint 用 historical close price 补跑（不是用当前价）
  - 同一时刻重叠运行被 DB advisory lock 拦截
- **真实端到端晶泰 2228.HK**：招股书时点跑分析 → 创建 snapshot → 状态机自动推进 → outcome_tracker 11 checkpoint → attribution → 生成 review_draft → 与人工事后判断对比 — 偏差在容忍范围内
- 所有 PROJECT_SPEC.md §3.11/§3.11.1/§3.11.2 / Phase 7.5 DONE 条件勾完
- 打 tag `v0.7.5`
- ADR 0012 Progress 4 行全部勾完

## Consequences

### Positive
- **每子阶段 1.5–2 天**，匹配 Phase 5/6/7 实证节奏 + 用户 Phase 工作流偏好（每阶段停下来等确认）
- **7.5a 一次性 18 表 + 4 trigger 全落地**，避免分两轮 Alembic migration 互相依赖
- **7.5b 顺手收掉 ADR 0011 遗留 2 条 Phase 7.5 Progress**，旧 ADR 可以完结
- **7.5c 状态机 + 业务规则隔离测试**，3 个仿真测试 + 30 家代码映射 + 5 家财报比对边界清晰
- **7.5d 端到端晶泰只跑一次**，不打散到各子阶段，避免 fixture 重复
- **延期路径无**：本 ADR 不引入延期项，因为 Phase 7.5 spec §3.11 是自包含的；Phase 8 backtest router 复用 7.5d 调度器框架但实装在 Phase 8

### Negative
- **7.5a 工作量集中**（18 表 + 4 trigger + 大量 ORM + Phase 7 in-memory PG 化）：单次 commit diff 大，难 review
  - **Mitigation**：3 个 Alembic migration 文件按 v1.1 / v1.2 / v1.2.1 拆，ORM 文件按 7 张表/组拆；同一 commit 但 diff 可分块 review
- **7.5d 端到端依赖 7.5c 全部完成**，临界路径不可并行
  - **Mitigation**：符合 spec DONE 条件；7.5c 早期就能跑独立单元测试，不卡 7.5b 进度
- **registry.py 从 in-memory 重写到 PG**：Phase 6 写的 89 行代码全废
  - **Mitigation**：Phase 6 in-memory 是 ADR 0010 明确标记的过渡方案，spec 早就预告 Phase 7.5 替换 PG，无返工损失

### Neutral
- ADR 0012 不引入 Phase 7.5 范围之外的延期项，与 ADR 0011 (Phase 7 切 MVP) 模式不同
- 子阶段命名沿用 7.5a/b/c/d 而非 8.0/8.1/...（不另开 Phase 编号），保持 spec Phase 编号一致
