# UI Integration

> Onboarding guide for the UI repo (`apps/web`). The backend is the
> authority — UI only renders and dispatches. Everything in this doc
> reflects an invariant the backend enforces; the UI just has to honour it.

---

## 1. Quick start

```bash
# Backend
cp .env.example .env                   # JWT_SECRET, DB url, KIMI key
docker compose up -d postgres qdrant   # services
uv sync --extra dev                    # python deps
uv run alembic upgrade head            # migrations
uv run uvicorn hk_ipo_agent.api.main:app --reload --port 8000

# UI (separate repo)
pnpm install
pnpm dev    # talks to http://localhost:8000
```

OpenAPI is at `http://localhost:8000/openapi.json`. Hit it to confirm
the backend is up + auth contract matches what the UI expects.

## 3. Type generation (CI gate)

```bash
# UI repo:
pnpm openapi-typescript http://localhost:8000/openapi.json \
  --output src/types/openapi.ts
```

CI runs this against the latest backend image and `git diff --exit-code`
on `openapi.ts`. A non-empty diff = the API changed and the UI has
work to do (rename a field, add a new endpoint binding, etc.).

This is the contract enforcement mechanism. Per CLAUDE.md §UI 集成约束:
> OpenAPI schema must auto-generate and be complete. Any API change
> must sync; the UI repo's `openapi-typescript` CI diff must be empty.

## 4. Decimal is a string on the wire (R5-6)

```typescript
// generated by openapi-typescript:
interface SnapshotOut {
  price_range_low: string;     // <-- not number
  total_cost_usd: string;      // <-- not number
  ...
}
```

JS `Number` is a 64-bit float and loses precision past the 17th
significant digit. Money is serialized as a JSON string and parsed
client-side with `Decimal.js` / `BigNumber.js` if you need to do math.
For pure display, just render the string — it's already formatted
to the right number of decimal places by the backend.

## 5. Errors: RFC 7807 only

```typescript
// always this shape — never `{error: "..."}`:
interface APIError {
  type: string;            // "about:blank" or a URN
  title: string;           // short human label
  status: number;          // duplicates HTTP status
  detail: string;          // multi-line OK for validation
  instance?: string;       // URL of the failing resource
  request_id?: string;     // correlate with backend logs
  validation_errors?: Array<{loc: string[]; msg: string; type: string}>;
}
```

UI error toast / inline error UX should key off `title` (for the
header) and `detail` (for the body). Include `request_id` in any bug
report — it correlates to the `X-Request-Id` header the backend
echoes on every response.

## 6. Pagination

```typescript
interface Paginated<T> {
  data: T[];
  meta: {
    total: number;     // overall count
    limit: number;
    offset: number;
    has_next: boolean; // simpler than computing from total
  };
}
```

Defaults: `limit=50`, `offset=0`, `limit ∈ [1, 500]`. Cursor pagination
is **not** used.

## 7. Auth flow

```
1. POST /api/auth/login {email, password}
   → {access_token, token_type: "bearer", expires_in: 3600}
2. UI stores in memory + httpOnly cookie (no localStorage — XSS risk)
3. Every fetch:  Authorization: Bearer <token>
4. Refresh: re-POST /api/auth/login before exp, replace token
5. 401 from any endpoint → redirect to login
```

For SSE / WS, the `Authorization` header doesn't work — use
`?token=<jwt>` instead. See SSE_PROTOCOL §2 + WS_PROTOCOL §2.

Phase 9 may add SSO (Okta / Azure AD) — see AUTH_INTEGRATION.md.

## 8. Live updates: SSE vs. polling

| Use case | Channel |
|---|---|
| Dashboard live tiles (status, latest snapshot, recent alerts) | SSE `dashboard.refresh` triggers a re-GET |
| Alert pop-ups | SSE `alert.created` directly carries enough payload |
| Snapshot list refresh after analysis | SSE `snapshot.created` |
| Analysis-run progress | SSE `scheduler.started` / `completed` |
| Anything else | REST polling (10-30s) |

The SSE bus does NOT replay — on reconnect, re-GET the relevant list
endpoint to reconcile.

## 9. Hiding UI per role

Don't trust client-side role checks for security (the backend always
enforces). But to avoid showing buttons that 403, the `GET /api/auth/me`
response returns the resolved permission set:

```json
{
  "id": "...",
  "email": "...",
  "roles": ["reviewer"],
  "permissions": ["snapshots.read", "...", "chat.use"]
}
```

UI pattern:

```typescript
const canAcceptProposal = me.permissions.includes("proposals.accept");
<Button disabled={!canAcceptProposal}>Accept</Button>
```

Cross-reference the matrix in [RBAC.md §3](RBAC.md).

## 10. Hard "do not" list (CLAUDE.md §UI 集成约束)

- ❌ Do not connect to PostgreSQL / Qdrant directly from the UI.
- ❌ Do not call the LLM from the UI (KIMI / Moonshot keys are
  server-side; cost guard wraps every call).
- ❌ Do not implement business logic / valuation math in the UI —
  the backend is authoritative.
- ❌ Do not invent a new error shape — RFC 7807 only.
- ❌ Do not store JWT in `localStorage` — use memory + httpOnly cookie.
- ❌ Do not poll the audit log faster than 5s — it's expensive.

## 11. CORS

`config/cors.yaml` lists the allowed origins per environment.
Production does NOT allow `*` — explicit allow-list only.

Adding a new UI origin = ADR + config change + redeploy.

---

## 12. Smoke tests for a new UI feature

When wiring a new screen:

1. Hit the endpoint with `curl` first — confirm shape matches
   `openapi-typescript` types.
2. Check the RBAC matrix — what perm does this need?
3. Check error shape — handle 401 / 403 / 422 / 5xx.
4. Check pagination — does it paginate? Default 50.
5. If realtime — confirm the SSE event type is in
   [SSE_PROTOCOL §4](SSE_PROTOCOL.md#4-registered-event-types).
6. Check `X-Request-Id` is captured in error logs.
